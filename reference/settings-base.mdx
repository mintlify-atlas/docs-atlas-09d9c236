This file is the root settings module for ElectroHome. `manage.py` loads it by default via `DJANGO_SETTINGS_MODULE=electrohome.settings`. The production settings file imports from this base and overrides specific values.

<Warning>
  Several values in this file are intentionally insecure for development. Do not deploy with `DEBUG = True`, the default `SECRET_KEY`, or an empty `ALLOWED_HOSTS`.
</Warning>

## Paths

### `BASE_DIR`

```python
BASE_DIR = Path(__file__).resolve().parent.parent
```

The absolute path to the project root directory (the folder containing `manage.py`). Used to construct paths for the database file, templates, and static files. Resolving two levels up from `settings.py` places `BASE_DIR` at the repository root.

## Core settings

### `SECRET_KEY`

```python
SECRET_KEY = 'django-insecure-j!&amsgcyeces&3lvzhfy*al7(zowo@227_y*$gt51bws(4lgm'
```

<Warning>
  The default value is a publicly known insecure key. You must replace it with a unique, unpredictable value in production. Generate one with:

  ```bash
  python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"
  ```

  Store the production key in an environment variable and never commit it to version control.
</Warning>

Used by Django for cryptographic signing of cookies, sessions, CSRF tokens, and password reset links. Any change to this value invalidates all existing signed data.

---

### `DEBUG`

```python
DEBUG = True
```

<Warning>
  `DEBUG = True` must never be used in production. It exposes full stack traces, local variables, and settings values in error pages.
</Warning>

When `True`, Django shows detailed error pages instead of generic 500/404 responses, enables the debug toolbar (if installed), and serves static files directly. The production settings override this to `False`.

---

### `ALLOWED_HOSTS`

```python
ALLOWED_HOSTS = []
```

An empty list is only valid when `DEBUG = True` — Django then accepts requests from `localhost` and `127.0.0.1`. In production this list must contain every hostname or IP address that can be used to reach the server.

Example production value: `['electrohome.example.com', 'www.electrohome.example.com']`

The production settings file reads this from the `ALLOWED_HOSTS` environment variable.

## Middleware

```python
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.common.CommonMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.messages.middleware.MessageMiddleware',
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
]
```

Middleware is applied in order on the way in and in reverse order on the way out.

| Middleware | Purpose |
|------------|---------|
| `SecurityMiddleware` | Enforces HTTPS redirects, HSTS, and other security headers. In production, Whitenoise static file serving is also added here. |
| `SessionMiddleware` | Manages session data using the configured session backend. Must appear before any middleware that uses sessions. |
| `CommonMiddleware` | Handles URL normalization (trailing slashes) and `DISALLOWED_USER_AGENTS`. |
| `CsrfViewMiddleware` | Validates CSRF tokens on `POST`, `PUT`, `PATCH`, and `DELETE` requests. |
| `AuthenticationMiddleware` | Attaches the authenticated user to each request object via `request.user`. |
| `MessageMiddleware` | Enables the flash messaging framework. Requires `SessionMiddleware`. |
| `XFrameOptionsMiddleware` | Adds the `X-Frame-Options` header to prevent clickjacking. |

## Database

```python
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.sqlite3',
        'NAME': BASE_DIR / 'db.sqlite3',
    }
}
```

The development database is SQLite, stored as `db.sqlite3` in the project root. No additional configuration or running database server is required.

<Note>
  SQLite is only suitable for local development and single-process testing. Production uses PostgreSQL, configured in `settings/production.py` via `dj_database_url`.
</Note>

## Authentication

### `AUTH_PASSWORD_VALIDATORS`

```python
AUTH_PASSWORD_VALIDATORS = [
    {'NAME': 'django.contrib.auth.password_validation.UserAttributeSimilarityValidator'},
    {'NAME': 'django.contrib.auth.password_validation.MinimumLengthValidator'},
    {'NAME': 'django.contrib.auth.password_validation.CommonPasswordValidator'},
    {'NAME': 'django.contrib.auth.password_validation.NumericPasswordValidator'},
]
```

Applied whenever a password is set or changed through Django's built-in forms and `manage.py createsuperuser`.

| Validator | Description |
|-----------|-------------|
| `UserAttributeSimilarityValidator` | Rejects passwords that closely resemble the user's username, first name, last name, or email. |
| `MinimumLengthValidator` | Requires at least 8 characters (Django default). |
| `CommonPasswordValidator` | Rejects passwords from a list of 20,000 common passwords. |
| `NumericPasswordValidator` | Rejects passwords that are entirely numeric. |

## Static files

### `STATIC_URL`

```python
STATIC_URL = 'static/'
```

The URL prefix for serving static files (CSS, JavaScript, images). In development, `django.contrib.staticfiles` serves files from each app's `static/` subdirectory at this path. In production, Whitenoise serves pre-collected files from `STATIC_ROOT`.

## Models

### `DEFAULT_AUTO_FIELD`

```python
DEFAULT_AUTO_FIELD = 'django.db.models.BigAutoField'
```

Sets the default primary key type for all models that don't explicitly define one. `BigAutoField` uses a 64-bit integer, avoiding the overflow risk of the older 32-bit `AutoField` for large datasets.
